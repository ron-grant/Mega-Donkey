Introduction

 

Mega Donkey is an Atmel ATMega128/2561 based microcontroller equipped with transflective graphics LCD available in both touch screen and non-touch screen models.

 

The Atmel ATMega series microcontrollers, containing the AVR RISC core, are powerful and efficient   machines.

 

If you are using a Windows platform, programming the donkey is accomplished via serial port connection from Donkey Prog – no programmer needed.  DonkeyProg is twice as fast as Atmel’s STK-500, speeding your program turn around time. For Example, 60K of code can be programmed into donkey in under 6 seconds on most machines.

 

If you are new to C, fear not, most all of our library functions  have simple interfaces. Also, many of our examples are not complex,  which will minimize time required to understand how they work and aid in learing the language. There  are many books covering C. The web is a great resource too. Including “C” in a search term, e.g. “C Language”  instantly produces valuable reference information. In fact, for that search the Wikipedia appeared to have a nice reference. Later you might find it helpful as a quick reference, e.g. “C Operator Precedence”.

 

Many of the Microcontroller I/O pins and device resources such as timers,  are not used for Donkey core operation, leaving a good deal of the microcontroller available for user programs and/or hardware interface. Also, while the donkey does do low-level graphics drawing, it is not relied upon to do low-level LCD refresh. This off-loads a burden that is present in some controller designs.

 

We wrote a robust set of library functions including, text, graphics, sound, serial I/O, I2C, hobby servo interface, motor encoder interface and more, to make sure that we not only provide you with a chunk of hardware, but you get some real value in custom code written to assist you in accomplishing your goals without spending countless hours to get the basics running.

 

Use the Variable Editor in your programs to get up in running with an interactive means of editing program parameters – OR - use Donkey Wiz to interactively build a menu framework and have all of the code generated automatically. Here, you write code to respond to button presses or to slider / check boxes, with everything else taken care of.

 

For editing C code and managing projects, one option is to use Atmel’s AVR Studio, which provides a seamless interface to WinAVR.  One advantage of AVR Studio is that it takes care of creating and maintaining a makefile for your project. For those new to C, the makefile is a script that directs the C compiler and linker to create an executable file that can be downloaded to your Mega Donkey.

 

Both WinAVR and AVR Studio, are free, and can be downloaded and installed easily on a Windows 2000/XP/Vista System.

 

Additional software including DonkeyProg, our serial programmer DonkeyTerm our LCD emulating graphics terminal (built into DonkeyProg application), and Donkey Wiz, our interactive menu designer are available for download from our website at www.mega-donkey.com

 

Atmel publishes extensive information, on the ATMega128 and ATMega2561 microcontrollers used in Mega Donkey. These are 300+ page specifications published as PDF files on their website at www.atmel.com . You can search directly for the parts on their website and be taken to a page with downloadable specifications.

 

Depending of the level of detail of your application you may not need to refer to the microcontroller documentation, but you might find at least skimming the documentation might give you some added  insights in using the microcontroller.

 

We found creating Mega Donkey to be an enjoyable task and continue to enjoy developing our own projects and sample applications for it.

 

We hope it helps you to realize your project goals.

Common Mega Donkey Hardware Features

·  Two RS-232 (Male = Data Terminal Equipment Interface) ports supported by hardware UARTs onboard the microcontroller  and  a Texas Instruments MAX213 RS-232 receiver/driver.

 

·  Serial Communication jumpers allowing allocation of any combination of signals to RS-232 Driver & Receiver for each port, allowing up to 8 processor I/O pins to be freed for other uses if not needed for serial I/O

 

·  ISP Connector for connection to serial programmer, e.g. Atmel STK-500

 

·  I2C Connector (two 5x1 pin headers available) for connection to I2C devices

 

·  JTAG Connector for connection to JTAG debugger/programmer

 

·  39 Pin (20 x 2  0.100" spaced male pins) header for interfacing to a wide variety of devices,  this connector is physically  compatible with cables used for old IDE hard drives (not pinout compatible)  which is handy for connecting your donkey to project boards where you want more space than is offered by the prototyping area

 

·  4 Low Power High Efficiency LEDs (software PWM intensity control available)

 

·  Mini 8-Ohm speaker supports variable pitch, and duration  tone generation 

 

·  Prototyping Area with lands for 8 pin SOIC and space available for up to a 40 pin DIP or several smaller devices

 

·  Power Input (via 2.1 mm center + jack) 7 to 12 VDC, with reverse polarity protection provided to prevent frying your donkey if you accidentally reverse the polarity

DonkeyProg – Serial Programmer & Mega Donkey Terminal

 

DonkeyProg is a Windows application that allows you to quickly program your Mega Donkey using its built-in serial bootloader. It also contains Mega Donkey Terminal which allows you to use your PC as a video terminal connected to your donkey with the ability to emulate / create a view of your Mega Donkeys  LCD screen and to react to touch screen inputs via mouse clicks.

 

DonkeyProg

 

Using DonkeyProg is straightforward. Its purpose is to allow selecting a .HEX file generated by compiling your program using, for example, the  AVR Studio IDE, then download that file to  your donkey via a serial data link. Also, DonkeyProg can be used to read/write your Donkeys EEPROM.

 

If you are getting started, we have provided a few sample .HEX files that you can try out independently of getting AVR Studio installed and up and running.

 

Cable Connections from PC to Donkey

 

You will need a COM port available on your PC and a serial cable. In most cases this will be a 9 pin cable with one female end that plugs into your PC and one male end the plugs into the null-modem adapter that came with your MegaDonkey. Alternatively you may have a Null-Modem cable which should be labeled as such, and would contain two female ends (if not labeled, you can try it, but it may not be wired correctly).

 

If your PC has no serial ports,  then you may need to buy a USB to serial adapter. These are available for about $15 to $20 from most computer stores or online. A native serial port is preferred, especially for connection to Donkey Terminal, but the adapters often work quite well.

 

Connect the Serial Cable to either COM Port on your Donkey, making sure that the RX and TX jumpers for that port are installed on the Donkey Serial Header (10x2 header). See connection diagram.

 

 

Power Up and Test Your Donkey

 

Make sure the serial cable is connected from PC to Donkey as described above, and power is connected to your donkey, then give the Donkey reset button, located next to the power jack,  a press. You should see the LEDs sequence on the Donkey. If not, refer to the Troubleshooting section .

 

Running Donkey Prog

 

Donkey Prog is a standalone program (DonkeyProg.EXE), needing no installation. It is located in your MegaDonkey folder in a folder named md_progs. You may wish to right click on DonkeyProg.EXE icon and create a shortcut, then drag that shortcut to the Desktop.

 

Clicking on DonkeyProg (or shortcut) starts the program.  It remembers any settings / file selections you make via a small DonkeyProg.ini file it writes in the folder where it resides, hence DonkeyProg does need to be located in a folder where data can be written, e.g. not floating around on a CD-ROM..

 

First, you must insure that the correct COM port is selected. If you are not certain which port your Donkey is connected to you might have to do some detective work. The easiest way to get a clue about what ports are connected to your computer is to consult the Windows Device Manager.

 

One way to access the Device manager (Win2K and XP) is click [Start] select Settings then select  control panel and click System icon. Then click the Hardware tab then press [Device Manager] button.then click on the Ports entry in the hierarchal list that appears.

 

Set the COM port on the DonkeyProg screen accordingly. Note that the connection to the port is not tested until you start the programming process. Then a message will report if connection the port was successful.

 

Next, (in DonkeyProg)  press [Browse] button to select a .HEX file to write to your donkey, using standard windows file navigation provided by the file open dialog that appears. Note that once you have selected a file, it remains selected which is handy since often you will be compiling the same program over and over and  downloading to your Donkey.

 

After selecting a file, the sure fire approach to catching your donkey with its boot loader running (the LEDs sequencing). Press and Hold donkeys reset button, then click  [Program] (and release) then release your donkeys reset buton.

 

Another option is to click [Program] and quickly click reset on donkey. This will often work, but may be less apt to work if your application program uses Mega Donkey Terminal. In this case, your donkey may be very chatty, causing problems with programmer if the bootloader is missed.

 

If you have difficulty downloading to your donkey, an option is to switch to the other Donkey port just to help rule out jumper issues on RS-232 header.

LCD  Text and Graphics (lcd.h)

 

To position text and graphics, the Donkey LCD screen uses a Cartesian coordinate grid where the upper-left corner is the origin (0,0), positive X to the right and positive Y down the screen.

 

Two constants are defined, COLS (columns) and ROWS, that describe the resolution or number of addressable pixels that your donkey LCD panel has.

 

MD-T1 & MD-T2       COLS=160,  ROWS=80

MD-A1                        COLS=128,  ROWS=64

 

The (X,Y) coordinates of the corners of the screen is thus:

 

(0,0) upper-left

(COLS-1,0) upper-right

(0,ROWS-1) lower-left

(COLS-1,ROWS-1) lower-right

 

If this is unfamiliar territory, see “md_simple_lcd.c” it might answer some questions.

 

Many of the LCD functions parameters use a datatype named COORD. On Mega Donkeys with resolutions below 256 ROWS and/or COLS this is defined as an “unsigned char”, which is simply a byte value (0..255). On Donkeys equipped with high resolution panels, COORD becomes a 16 bit integer. Example code will show you how you can define variables with this datatype. Using COORD datatype allows greater portability among Donkeys and can be a space saver for variables on Donkeys with sub 256 pixel ROW and COL resolution.

 

The  LCD screen is initialized by calling md_init() at the start of your main() function which in turn calls lcd_init() to initialize the LCD.

 

A second feature of md_init() is to take care of directing output from printf()  calls to the LCD. There may be occasions where you  might redirect  printf() output to a serial port, but then wish to resume output to the LCD. In this case  lcd_set_stdout() should be called.

 

You will find all of the lcd function prototypes in lcd.h. They are presented here in a compact form without the formal C syntax for the sake of  readability. If  not otherwise mentioned, all variables are of type COORD.

 

You will find simple demo of these function calls in “md_simple_lcd.c”.

A more complex and interesting demo can be found in “md_demolcd.c”.

 

A new feature (Feb 2009) is the initialization function, md_init_no_logo(). It performs the same duties as md_init, except it eliminates the display of the donkey logo and library version number on reset allowing for a faster system reset.

Common LCD Functions (lcd.h)

 

lcd_clear()                                clear the screen, all the pixels turn off, which depending on your LCD can

mean different things. On MD-Tn and MD-An models, your screen goes light. Also, as a side-effect of the function, color is set to 0xFF (WHITE) which is really black. So if you follow lcd_clear() with  line(0,0,100,50) you will see a black line even though color = WHITE.

 

set_color(c)                              sets color for characters and figure drawing (lines,dots,ellipses…)

set_bg(c)                                  sets background color

 

set_charsize(s)                          sets  char_size which is integer multiple scale 1x,2x,3x

 

lcd_set_stdout()                        setup LCD so that printf() will output to it , this need not be called unless

you redirect to a serial port , e.g. com0_set_stdout()

!!! verify com stdout function name

Note: that lcd_char and lcd_text functions described below, ALWAYS output to the LCD regardless of  where stdout (standard output) is directed – so this function call does not apply to them.

 

lcd_setxy(x,y)                           define (x,y) position of upper-left corner of next character to be drawn by

                                                printf().

 

lcd_char(x,y,c)                        draw a character at (x,y) where c is type unsigned char (0..255) where 

                                                (x,y) is coordinate of upper-left corner of character

 

lcd_text(x,y,*s)                         draw a null terminated string starting at (x,y) where *s points to null terminated string (e.g.  lcd_text(0,0,”Hello World”);  x,y specify  upper left corner of first character in string.

 

lcd_textPS(x,y,ps)                 same as lcd_text except the string you specify is never copied into RAM

                                                e.g. lcd_textPM(0,0,”Hello World and this saves 43 bytes of RAM”);

LCD Bit Blitting

 

The function blit (blit stands for Block Transfer) is used copy one rectangular region of the screen to another and also to perform a bit-by-bit logical operation on two rectangular regions then store the result in the second region we call the destination region whose upper-left corner is located at (x,y) location (dest_left,dest_top).

 

blit(left,top,right,bot, dest_left,dest_top)  

 

For example logical OR of two regions would include pixels set in either or both regions in the destination region.

 

Note that left,top is the upper-left corner (x,y) of the source rectangle and right,bot is the lower-right corner of the source rectangle. Since the destination rectangle is the same size as the source, only the upper left corner of the destination is supplied (dest_left,dest_top).

 

 

Use function set_draw_flags(n)  set to one of the following values before calling the blit function.

           

            0                                  source region copied  to destination region

 BLIT_OR                   source region is ORed with destination region and written                     

 BLIT_AND               source region is ANDed with destination region and written

 BLIT_XOR               source region is Exclusive ORed with destination region and written

 

Special modes operating on a single line (1 pixel high ) using an array called blit_buf [] for either source or destination. blit_buf[] stores 8 pixels per byte on donkeys equipped with monochrome LCDs

 

blit_buf[] is sized large enough to hande a scanline + 2 bytes, e.g. on MD-T1/T2 blit_buf is defined as

u08 blit_buf[20+2].

 

 

BLIT_WRITE          OR one of the above with this to use  blit_buf [ ]  as source

BLIT_READ            OR one of the above logical draw_flags  to used  blit_buf[] as a destination  BLIT_ALIGNED     OR in this this flag before doing a BLIT_READ or BLIT_WRITE if  blit_buf[] contents already aligned with screen

Menu Functions (menu.h)

 

This section describes the variables and functions used to create menu screens that Donkey Wiz automatically generates from menus you lay out interactively.

 

The key points to using  the mega donkey menu system.

 

  • Each Menu is implemented as a function.

 

  • User code may be placed throughout menu, but keep in mind Donkey Wiz generates empty code framework so if you have many changes you want to make to a menu using that tool, you will have to add your code back in.

 

  • “MENU_INIT  do {      } while (menu_cmd() != MENU_EXITCODE);” forms the body of a menu loop that is executed until user presses exit button, provided that an exit button definition is included in the menu body OR you assign a response code = MENU_EXITCODE  (255) to one of your controls, i.e. a button most likely.

 

  • Within the body a macro MENU_CONTROLS appears following which is a list of menu control functions that describe the buttons, checkboxes, slider controls, label text, and boxes that may appear in the menu.

 

  • The macro MENU_COMMANDS appears at the end  of the menu controls. Its purpose is to inform the menu system to only draw the controls one time, then update them as needed.

 

  • Typically a switch statement is placed in the body of the loop to handle responses to control interaction based on response code generated by each control, e.g. if a button is pressed its response code (bytre) is returned by menu_cmd() called in switch statement.

 

  • Examples are the best way to learn the menu system. Also, creating a few simple menus with Donkey Wiz is the way to go since it writes the menu code for you.

 

 

Note that Both Checkbox and Slider controls are implemented as a combination of a function call and a macro to allow passing a parameter by value and returning a result implicitly. For example consider a function that modifies a variable could be written as  x = func(x) or as func(&x) where parameter is passed by address (reference). Here we implement func(x) where really we are performing x=func(x) via macro.

Analog Input / Touch Screen Support (adc.h)

 

The Mega128/2561 microcontroller provides 8 analog input channels, two of which are used by the touch screen on touch screen Mega Donkey models.

 

Under normal operation the Mega Donkey system software takes care of the process of scanning the number of channels you specify by calling in adc_init(Channels) OR by default all 8 channels. The rate of the scan is 5,000 samples per second in round robin channel scan one channel at a time,  that is each channel is updated at a rate of 625 samples per second.

 

You can, however, disable the system software, giving you complete control over the controller’s ADC hardware, including, if needed, the ADC interrupt vector, which is not used by the system software. In doing so, you lose touch screen support. With at least occasional sampling of one of the touch screen channels it would be possible to detect a touch to stop a process then re initialize the touch screen via call to adc_init function.

Executing a Functions Periodically Under System Timer Control

 

You can link your own functions that need to be called periodically into the system timer by using the method illustrated in demo program included below.

See also: md_Periodic_Int demo project in md_demos folder.

 

NOTE: Requires libdonkey128_103 or libdonkey2561_103 or later. If you library file version is less than 1.03 (library named libdonkey128.a or libdonkey2561.a) then this feature will not work.

 

Provisions exist for calling user functions at 100Hz (100 times per second) or 10KHz (10,000 times per second). Note that your function, in effect, becomes part of the timer0 interrupt service routine, therefore attempting to print to LCD would be discouraged (!!! if even possible?)

 

Another method to call user routines outside of interrupts can be accomplished by using get_time_alive  See md_XXX.c  !!!

 

The timer0 module defines a pointer named “service_100Hz” which is a pointer to a function. Initially the pointer is NULL. 100 times per second, the timer will call the function pointed to by “service_100Hz” if it is not NULL. If we assign the address of a function we want called to the variable (while interrupts disabled)  then everything works great. For example:

 

cli();

service_100Hz = my_function;  // where my_function is defined as void my_function(void) { … }

 sei();

 

Now our function “my_function” is called at 100 Hz. Only one small detail to consider now: If there was already a function assigned to service_100Hz then we just replaced it and it won’t be called.

 

The solution is to save the address of the function we are about to replace then call that function after our function is finished. To do this we must declare a pointer to function variable then copy the address of the function (stored in service_100Hz) to the variable and call it when our function finishes.

 

If no function was being called, then service_100Hz would be NULL, in which case we would not need to store it, but we do anyway to make code simpler.  We do however need to test to see if the pointer is NULL before trying to call the function.

 

Note that code would be identical for 10KHz interrupt, except replace “100Hz” with “10KHz” throughout.

 

 

Example of linking a user 100Hz interrupt routine into chain of routines

 

See Discussion in previous section.

 

void (*after_my_100Hz)(void) = NULL;  // pointer to next service routine in chain

 

void my_100Hz_update(void)  // function to be linked into chain of routines running at 100Hz

{

  // user code here (should not be too time consumptive!)

  // this is being called from an interrupt routine, hence should use care in doing I/O

  // don’t allow  early “return” must execute code at bottom of this funciton

 

  speaker_tick();  // simple test

 

 

   // call next routine in timer chain ONLY if defined (not NULL)   

    if(after_my_100Hz)  after_my_100Hz();

}

 

 

void init_my100Hz_demo(void)

{

     u08 sreg;

 

     sreg = SREG;                      // save status register

     cli();                                    // disable interrupts

 

     after_my_100Hz = service_100Hz;       // save pointer to old service routine to be called after

                                                                    // our routine finishes

     service_100Hz = my_100Hz_update;   // pointer to our new service routine – timer0 service function

                                                                    // is not set to call my_100Hz_update 100 times per sec.

     SREG = sreg;                                          // re enable interrupts

 

 

     // my_100Hz_update is now being called 100 times per sec. provided that interrupts are enabled

    

  }

Serial Communicaitons (uart.h)

 

Mega Donkey contains two serial ports  supported by hardware UARTs on the Mega128/2561. Each port is buffered through an RS-232 transceiver and uses standard DB9 (technically DE9) connector with DTE (Data Terminal Equipment) configuration (Male Pins) same as a typical PC serial port.

 

Note that a null modem adapter, included with your Mega-Donkey,  is required to interface the Donkey serial port to your PCs serial port – provided that your are not using a null modem cable which eliminates the need for the adapter. Such a cable would have two female connectors and would hopefully be labeled “Null Modem”.

 

Transmitted and received characters both use 120 byte (character) FIFO (First In First Out) queues.

Donkey supports 6,7 or 8 bit characters. UART does support 9, but donkey does not.

Variable Editor

 

The variable editor provides an interactive tool that allows modifying program parameters (variables) at runtime with a minimum of effort.  It allows you to edit  8,16 or 32 bit integer variables and also control  bit switches within 8,16 or 32 bit variables. 

 

The variables to be edited are specified by a function you define which includes labels to help identify the variables. Also, the variables can be grouped to allow quick navigation from group to group.

 

With the variable list function declared you need only initialize the variable editor passing your list function then repetitively call ve_update() until it returns false which happens when the user closes the variable editor via a press of the exit button [x].  (See example code at the end of this section – also example program md_VarEditor .c)

 

While the editor is running  the following commands buttons are available:

 

[G]       Select the next group

[>>]     Select the next variable within a group

[<<]     Select the previous variable within a group

[+]       Increment Integer variable (or set bit variable to 1)

[-]        Decrement Integer variable (or set bit variable to 0)

[#]        Direct Numerical entry using Donkulator keypad

[W]      Write Variables to EEPROM

[R]       Read Variables back from EEPROM

[X]       Exit the Variable Editor

 

Variable Editor - Read / Write Variables from / to EEPROM

 

Reading and Writing Variables can also be accomplished under program control via read and write functions (where display =1, if you want to display variables as they are read/written)

 

ve_read(display)

ve_write(display)

 

Note that if you change what variables you include in your variable list or the order they are specified, your saved data (if any written) will not be valid, and will need to be rewritten, before any read operation. You can change group names and variable label names since they are not stored in EEPROM.

 

Variable Editor – Redirecting to Serial Port

 

Redirecting the Variable Editor I/O to one of the serial ports is also possible. After calling ve_init, call ve_setport with device = serial port number  0,1…  This will allow you to use a video terminal connection to manipulate the variable editor.  This differs from using the Mega Donkey Terminal which provides emulation of the LCD touch screen.

Donkey Wiz – Menu Designer (Windows Application Program)

 

Donkey Wiz is a tool that lets you interactively layout menu screens, including buttons, checkboxes, slider controls and boxes then generates a C code framework that realizes the basic functionality of your menus. This includes menu to menu navigation via button presses, and maintaining variables operated on by sliders and checkboxes.

 

Donkey Wiz provides the ability to generate a main function, producing a working application program that you can compile, download and execute, before you write code to respond to button press events or control changes.

 

Each menu screen runs a loop which allows you to add code that updates realtime processes that are not serviced by interrupts.

 

When Donkey Wiz starts up, it creates a main menu named “menu_main” with a title label “MAIN” centered on the top line. Also, the menu is set to not have an exit button ([X] in upper right). Note that you can edit  all of these these selections – discussed later.

 

Components

 

The components available for placement on your menu screens are show in a tool bar at the top of the Donkey Wiz  main window (to the right of  File New,Load and Save Icons). Note there is a hint for each button – passing the cursor over the button will display a description of the given icon.

 

To Add components to the LCD screen area (large pale blue box), click on a component in the component palette then click in the LCD screen area where you want the component (upper-left corner) placed. 

 

 

Label  - Creates a static text label

 

Button – Creates a button which generates a response code that is decoded in a switch statement. Optionally a menu call can be associated with the response code that generates code to call up a given menu, whereupon exiting that menu will return control back to the calling menu.

 

Checkbox – check box that defines a byte variable (u08 = unsigned char) which will be set to 0 when box is unchecked and 1 when box is checked. Also checkbox contains a textual label which can be defined or cleared.

 

ButtonRow – creates a row of buttons where each character in the ButtonsLabels field becomes a button with the exception of spaces “ “ which are used to generate horizontal spacing. Note that all the button row (and button col) letters must be unique on a given menu.

 

ButtonCol – same as button row, buttons arranged in a  vertical column.

 

Horizontal Slider – Slider control that defines an integer variable, global to the module,  of (8, 16 or 32 bits by choice of  VarType) and VarName. Also, MinValue and MaxValue are defined which limit the range of the variable.

 

  • Donkey Wiz 1.0 does not perform range checks for a given  VarType. That is you must insure your limits fall within the ranges listed below.

 

  • Control Limits (for 32 bits integers) are approx 1,000,000 to -1,000,000 due to limitations in 32 bit arithmetic used to process controls.

 

 

 Extremes for Limits  (MinVal…MaxValue)

 

 u08     unsigned 8 bit numbers  (0..255)

 s08      signed 8 bit number (-128..127)

 word   unsigned 16 bit integer (0..65536)

 int        signed 16 bit integer (-32768…32767)

 long     signed 32 bit integer (-1000000…1000000)  (control limitation – not arithmetic limitation)

 

unsigned long  (0…1000000)

 

Optionally a caption may be defined for the control (is defined by default, e.g. “hs1”, “hs2”...) and a checkbox indicates if the control will display a numerical value.

 

A response code is generated which can be tied to the menu’s case statement, but is not by default.

 

 

Editing Components

 

Clicking on a component and holding left mouse button down allows component to be dragged.

 

Resizing Box component can be done by clicking inside the lower-right corner of the box and dragging.

 

Slider width (or Height for vertical sliders) is edited by dragging the right (bottom for vertical slider).

 

Clicking on a component selects it for editing. A yellow box appears around the selected component.

You can then edit fields relating to the component or delete it  by pressing [Delete Component].

 

You can choose to have Donkey Wiz ignore boxes when editing by checking “Edit Ignore Boxes” . This will make it easy to select components nested within a given box.

To add additional menus, press [Add Menu] to add a new menu. Each menu creates a C function that includes all of the code to draw the menu and update its controls.  By default, a name prefix of “menu_” is applied to menus you create. For example if you add a menu “calibrate”,  a menu function called “menu_calibrate” will be created.

Buttons on any given menu can be setup to call any other menu that is defined. This is accomplished by selecting the button then clicking on its “Menu Call” field value. A drop-down list appears. Click on the menu you wish the button to call.  At present you will need to download menu code to the Donkey to verify your menu call links.

Troubleshooting

 

Donkey Power Up

 

First, question is: In there anything on the LCD screen? If so, you have power. If you can’t program the donkey read on:

 

Note: You might have power if LCD is blank. Also, on Donkeys equipped with contrast POT, you may need to adjust the contrast setting. Also, if you have previously programmed your donkey and you are not calling md_init(); first in your program, or at all, then you might have a case where you are not initializing the LCD or displaying anything.

 

If you have not programmed your donkey, note that it is shipped with a series of demo programs which will activate when you first power up your donkey.

 

Make Sure Donkey LEDs sequence on power up (unless ISP Jumpered)

 

Upon Power Up of Donkey (plugging into power adapter) , the  LEDs should blink in sequence  one time (over about a 2 second time period) while the bootloader runs looking for a new program to be downloaded.

 

If such is not the case double check that there is not a jumper installed on ISP header from pins 7 to 8 (pulldown PB1)  which will cause the bootloader to quickly  stop with no LED sequence.

 

If you are using a jumper to inhibit boot sequence, you might want to temporarily remove it to help verify that your Donkey is getting power.

 

At this point if you are not seeing the blink sequence it is possible power is not getting to your donkey.

 

Checking To See If Your Donkey Is Getting Power

 

Check your power adapter connections and make sure it is in fact, plugged in to a good plug.

 

Remember that if your adapter has the wrong polarity, donkey will  not work. You need center the center  to be (+) and outside of connector to be (-) and adapter voltage needs to be 7-12 VDC, with current rating above 200 mA.

 

The LED sequence is generated by donkey’s bootloader which is a small program that is protected from erasure except in cases where an ISP programmer like Atmel’s STK-500 has been used to program your donkey – generally speaking, you should never need this programmer, hence you should not be running into problems erasing your bootloader.

 

If  for some reason you don’t get the LED sequence, first, make sure everything is disconnected from your donkey other than power, just to rule out any add on circuits and/or cables are not shorted, then if  you have a voltmeter, you can check that you have power getting to the board.  Testing the voltage at the voltage regulator is the easiest. The heatsink screw is GND and the side pins are Input (greater 7 volts or more) and  Output (5 volts).  The PC Board is labeled. If that checks out then you might double check pressing reset, after that you might want to try another adapter – making sure it is DC and not AC.
